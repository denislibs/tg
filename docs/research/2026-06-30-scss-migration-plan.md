# Полный план миграции UI на SCSS (build-step) + уход от MUI

Дата: 2026-06-30. Базовый разбор и обоснование выбора — в [`../ui-kit-migration.md`](../ui-kit-migration.md).
Решение зафиксировано: **SCSS компилируется на сборке (Vite + `sass`), рантайм-стоимость нулевая**;
стили берём 1:1 из tweb; уходим от рантайм-CSS-in-JS (emotion `sx`) постепенно (strangler).

## Принципы (обязательные, приоритет над «удобством»)

1. **Стили tweb — основа.** Источник истины — `tweb/src/scss/**`. Разметку, классы, размеры, токены,
   кейфреймы берём **1:1 из tweb**, не выдумываем. Перед «так в Telegram» — открыть `.scss` tweb.
2. **CSS-first: если что-то можно сделать на CSS — делаем на CSS, не на JS.** JS остаётся только там,
   где без него нельзя (данные, сеть, роутинг, реально императивные жесты/спринги). Состояния, ховеры,
   адаптив, простые анимации, переключение темы, видимость, скролл-эффекты — на CSS. Это и быстрее tweb,
   и совпадает с тем, как сделан сам tweb (он почти не использует JS для стиля/анимаций).
3. **MUI выпиливаем полностью.** Цель не «сократить», а **убрать `@mui/material` и `@emotion` из зависимостей**.
   Никаких «холодных экранов на MUI» как постоянного состояния — MUI это строительные леса на время миграции.
4. **Никакого рантайм-CSS-in-JS.** Ни `sx`, ни `styled`, ни инлайновых объектов стилей. Динамика —
   только через CSS-переменные на элементе (`style={{'--x': value}}`) + `var(--x)` в классе.

## Что обязано уехать из JS в CSS (инвентарь)

- **Тема**: JS-объект `tg`/`ThemeProvider`/`useTheme` (~83) → CSS-vars + `data-theme` на `<html>`. Ноль ре-рендеров.
- **`mode === 'dark' ? a : b`** ветки → отдельные `--tg-*` токены на тему, читаем `var()`. Ноль JS-ветвления по теме.
- **`useMediaQuery` (5)** → CSS media/container queries. Адаптив панелей (напр. `narrow` в `UserInfoPanel`) →
  container query / `@media`, без JS-состояния.
- **Ховеры/фокус/нажатие/disabled** → `:hover` / `:focus-visible` / `:active` / `[disabled]` (+ `color-mix()`
  для производных оттенков), не `useState`/`onMouseEnter`.
- **Условный стиль по состоянию** → `[data-state=…]`, `:has()`, `:nth-child`, аттрибуты — вместо JS-классов-флагов.
- **Скролл-эффекты** → scroll-driven animations (`animation-timeline: scroll()`) + `@property`. Fold историй
  уже так сделан (`--stories-fold`) — это эталон, расширяем подход.
- **Простые enter/exit/переходы** → CSS `transition` / `@keyframes` / `@starting-style` / `transition-behavior:
  allow-discrete` / (где уместно) View Transitions. **framer-motion оставляем только** для того, что CSS не
  тянет: drag/жесты, спринг-физика, layout/FLIP. tweb эти простые случаи уже делает на CSS — берём 1:1.

## Фаза 0 — Фундамент: Sass + токены в CSS-vars (без визуальных изменений)

Цель: подключить препроцессор и перевести тему на CSS-переменные так, чтобы существующий код
не менялся, а смена темы перестала ре-рендерить дерево. Это разблокирует всё остальное.

Шаги:
1. `npm i -D sass`. Vite подхватывает `.scss` без конфига. (Опц. `css.preprocessorOptions.scss.additionalData`
   для авто-`@use` общего `_mixins`/`_tokens` — добавим в фазе 1, не сейчас.)
2. Создать `src/styles/`:
   - `_tokens.scss` — все 38 токенов как `--tg-*` в `:root` (= пресет `classic`, дефолт) и оверрайды
     в `[data-theme="day"|"night"|"dark"]`. `bgGradient` (массив из 4) → `--tg-bg-grad-0..3`.
     Значения копируем **дословно** из `theme.ts` (classic/day/night/dark уже посчитаны).
   - `index.scss` — `@use 'tokens';` + перенос содержимого текущего `index.css` (кейфреймы typing уже 1:1 tweb).
3. Импортировать `./styles/index.scss` в `src/main.tsx` (вместо/рядом с `index.css`; затем удалить `index.css`).
4. **Алиас `tg.*` → `var(--tg-*)`** в `theme.ts`: заменить литералы в `TgTokens`-объектах на строки
   `'var(--tg-accent)'` и т.д. Один объект `tgVars: TgTokens` (значения = `var(--tg-*)`), он же идёт в
   `createTheme({ tg })`. Существующие `sx={{ color: tg.textPrimary }}` начинают эмитить `var(--tg-textPrimary)`.
   - `bgGradient` остаётся массивом, но из `['var(--tg-bg-grad-0)', …]`.
5. **Применение темы атрибутом**: в `App.tsx` `useEffect` ставит `document.documentElement.dataset.theme = preset`
   (`resolvePreset(themeChoice)`), слушает `prefers-color-scheme` для `system`. ThemeProvider/`buildTheme`
   **остаётся** для палитры MUI и `CssBaseline` (скроллбары), но его `tg` теперь var-based.
6. **`palette.mode`-ветки**: найти места `mode === 'dark' ? a : b` (напр. `UserInfoPanel.cardBg`,
   scrollbar в `buildTheme`). Перевести значения в отдельные `--tg-*` токены (свои на тему) и читать `var()`,
   чтобы убрать JS-ветвление по теме. Скроллбар-цвета — в `_tokens.scss`.

Подводный камень (задокументировать, не блокер): MUI считает `contrastText` из `palette.primary.main`;
если туда уехал `var(--tg-accent)`, расчёт контраста сломается. Митигация — в `buildTheme` для
`palette.primary.main` оставить **конкретный** accent пресета (не var), а `tg.accent` (для нашего кода) —
var. MUI-компонентов мало (≤6 инстансов), риск точечный.

Критерий выхода фазы 0:
- Переключение темы меняет только `data-theme`, дерево React **не ре-рендерится** под смену цветов
  (проверить react-scan: 0 рендеров компонентов при toggle).
- Пиксель-парити: все 4 пресета визуально идентичны до/после (скриншот-диф ключевых экранов).
- `tsc -b` чист, тесты зелёные, `vite build` ок, бандл не вырос.

## Можно ли «скопировать всю папку scss tweb»? — Нет (разбор)

tweb `src/scss` = **728 КБ, 272 файла**. Делится на слои с разной переносимостью:

1. **Фундамент (DOM-независимый) — копируем 1:1.** `variables.scss` (SCSS-консты:
   радиусы, брейкпоинты, размеры/высоты), `functions.scss`, чистые миксины
   `respondTo`/`textOverflow`/`textCenter`. Эмитят 0 CSS до использования.
   **СДЕЛАНО:** перенесены в `src/styles/{_variables,_functions,mixins/*}.scss` +
   единая точка `_foundation.scss` (`@use '../styles/foundation' as *`).
2. **Миксины, завязанные на runtime-классы tweb — адаптируем при использовании.**
   `_hover.scss` (`html.no-touch &:hover`, `:global(.night)`), `_animationLevel.scss`
   (`body.animation-level-2`), `_premium.scss` (`body.is-premium`), `_splitColor.scss`
   (своя система `--light-*/--dark-*`). Дословный перенос даст селекторы, которые
   у нас ни с чем не совпадут. План: `hover` → современный `@media (hover: hover)`;
   `.night` → наш `[data-theme]`; splitColor — у нас тема уже в `--tg-*`.
3. **`partials/` (81+ файлов, ~600 КБ) — НЕ копируем массово.** `_chat.scss`,
   `_chatBubble.scss`, `_chatlist.scss`, `_leftSidebar.scss` и т.д. стилизуют DOM/классы
   tweb (Solid). Без совпадающей разметки это мёртвый CSS + раздувание бандла (против
   цели). **Приносим правила 1:1 ПО КОМПОНЕНТУ** в его `.module.scss` в момент
   конвертации, когда есть совпадающий DOM. `base.scss` (2441 стр.) — туда же:
   глобальный токен-блок берём, остальное (базовые стили под DOM tweb) — по мере нужды.

Итог: «то же, что у tweb», но в правильной последовательности — иначе 600 КБ CSS,
красящего чужой DOM.

## Фаза 3 — Конвертация горячих путей (strangler, по компоненту)

Цель: снять per-render emotion-налог там, где он реально бьёт по fps. Порядок строго по «горячести».

Целевые компоненты (в порядке):
1. `components/ChatListItem.tsx` (рендерится для каждого диалога, часто).
2. `components/messages/MessageRow.tsx` + `MessageBubbles.tsx` + `RealMediaBubble.tsx` (строка сообщения,
   ~11 Box на строку — главный источник налога).
3. Хрома сайдбара: `Sidebar` header/search, `ChatHeader`.

Рецепт на компонент:
- `<Box sx={{…}}>` → `<div className={s.x}>` + правило в `Component.module.scss` (значения через `var(--tg-*)`).
- `<Box component={motion.div} sx={{…}}>` → `<motion.div className={s.x}>` (motion-пропсы остаются,
  только стиль уезжает в класс). Это 73 точки суммарно — но на горячих путях их немного; правим адресно.
- `<Typography sx>` → текст-классы из фазы 2.
- Статические стили — в `.module.scss`; динамические (зависят от пропсов/состояния) — через инлайн
  `style={{ '--x': value }}` + `var(--x)` в классе (CSS-переменные на элементе), **не** через `sx`.
- Замер `react-scan` до/после на каждом шаге; парити-скриншот.

Критерий выхода фазы: на профиле прокрутки списка/ленты emotion-работа на горячих путях ≈ 0;
визуально без регрессий; framer-motion-анимации (бабблы, лэддер, оверлеи) работают как раньше.

## Сквозные правила и риски

- **1:1 tweb**: стили целевых компонентов брать из соответствующих `tweb/src/scss/partials/*.scss`,
  не выдумывать. Перед «так в Telegram» — открыть исходник.
- **Динамика без sx**: значения, зависящие от пропсов/рантайма — только через CSS-переменные на элементе
  (`style={{'--x':…}}`), статика — в `.module.scss`.
- **framer-motion** — на горячих путях, где tweb делает анимацию на CSS, переводим на CSS 1:1
  (`@keyframes`/`transition`/`@starting-style`), а не сохраняем JS-анимацию. framer-motion остаётся
  только для drag/жестов/спринга/layout. Где пока оставляем motion — меняем носитель стиля (`sx`→`className`).
- **Сосуществование MUI и SCSS** во время миграции разруливается `@layer` (наши классы в `components`/
  `utilities`, MUI — вне слоёв = выше по специфичности, поэтому конвертируемый компонент полностью
  уходит с MUI за один заход, без «полу-стилей»).
- **Проверка на каждом шаге**: `tsc -b` + `npm test` + `vite build` + скриншот-парити (4 темы) +
  `react-scan` на горячем пути. Откат — пофайловый (миграция изолирована по компонентам).
- **Главный риск** — регрессии анимаций на узле `motion.*` и расчёт контраста MUI (см. фазу 0). Оба
  локализованы и покрыты критериями выхода.

## Минимальный первый коммит (старт фазы 0)
`npm i -D sass` → `src/styles/_tokens.scss` (38×4 токена) + `index.scss` → импорт в `main.tsx` →
`tg.*` = `var(--tg-*)` в `theme.ts` → `data-theme` на `<html>` в `App.tsx`. Без конвертации компонентов.
Это уже даёт: смену темы без ре-рендеров и фундамент под весь остальной план.
