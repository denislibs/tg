## Проблема

Сейчас `ConversationView` рендерит **все** сообщения из `win.msgs` (`src/core/hooks/useMessageWindow.ts`) — никакой виртуализации/обрезки нет. При долгом скролле вверх (`loadOlder` подгружает страницы по 40) окно растёт неограниченно:

- DOM раздувается → каждый reflow/ResizeObserver-коррекция дороже;
- память на тяжёлых медиа-нодах (`RealMediaBubble`, видео/картинки) не освобождается;
- на текущих объёмах диалогов это незаметно, поэтому помечено как оптимизация, а не баг.

tweb это решает обрезкой окна: держит в DOM ограниченное число сообщений вокруг вьюпорта, далёкие страницы выкидывает и при скролле обратно подгружает заново.

## Как это делает tweb

- Хранит видимый диапазон сообщений; при выходе страницы далеко за пределы вьюпорта она удаляется из DOM.
- Любое изменение содержимого (cull, prepend, append) оборачивается в **ScrollSaver** — фиксирует якорь до мутации и восстанавливает `scrollTop` после, так что визуально ничего не прыгает.
- Скролл к удалённому краю → реакция реальной подгрузки (как наш `loadOlder` / `loadNewer`).

## Что хочу сделать

1. **Ограничить окно в `useMessageWindow`**
   - Ввести `MAX_WINDOW` (напр. ~120 сообщений, кратно `limit=40` → ~3 страницы).
   - При `loadOlder`, если после добавления `msgs.length > MAX_WINDOW`, **обрезать хвост снизу** (самые новые), выставив `reachedBottom = false`.
   - При `loadNewer` (он уже есть в API: `loadNewer`, `reachedBottom`) — симметрично обрезать сверху, выставив `reachedTop = false`.
   - Обрезка строго по id/seq, ключи остаются стабильными (`m-${id}`), так что React просто размонтирует выпавшие ноды.

2. **Сохранение позиции при обрезке (ScrollSaver-аналог)**
   - В `ConversationView` уже есть единственная точка коррекции — `correctScroll()` + `pendingRestore` (distance-from-bottom) + `useLayoutEffect([win.msgs])`.
   - Для обрезки **сверху** (при скролле вниз) этого механизма мало: меняется высота над вьюпортом → нужен якорь «расстояние от верха» либо привязка к конкретной ноде. Надо обобщить `pendingRestore` до двух режимов: hold-from-bottom (есть) и hold-from-top (новый, для top-cull).
   - Обрезка **снизу** при скролле вверх позицию вьюпорта не трогает (режется то, что ниже экрана) — достаточно не сбивать `pendingRestore`.

3. **Перезагрузка при возврате**
   - Скролл к низу после bottom-cull → `reachedBottom === false` → существующий обработчик у нижнего порога зовёт `loadNewer()`.
   - Скролл к верху → как сейчас, `loadOlder()`.
   - Проверить, что флаги `reachedTop/reachedBottom` корректно ведут себя после cull (сейчас `loadNewer` есть, но `reachedBottom` по умолчанию выставляется при первичной загрузке — нужно убедиться, что cull его сбрасывает).

## Затрагиваемые файлы

- `src/core/hooks/useMessageWindow.ts` — логика обрезки в `loadOlder`/`loadNewer`, новый `MAX_WINDOW`, сброс `reachedTop`/`reachedBottom`.
- `src/components/ConversationView.tsx` — обобщить `correctScroll`/`pendingRestore` до режима hold-from-top; убедиться, что нижний порог триггерит `loadNewer`.
- `src/core/hooks/useMessageWindow.test.ts` — тесты на обрезку (после loadOlder длина ≤ MAX_WINDOW, reachedBottom сброшен; loadNewer восстанавливает хвост).

## Критерии готовности

- DOM-нод сообщений не больше ~`MAX_WINDOW` при сколь угодно долгом скролле в обе стороны.
- Ни при обрезке, ни при последующей дозагрузке вьюпорт не прыгает (визуально плавно, как в телеге).
- Скролл вверх→вниз→вверх по длинной истории работает без потери позиции и без дублей сообщений.
- `tsc` чистый, vitest зелёный, проверка на verify-стенде (`:38080`).

## Риски / на что смотреть

- Двойная коррекция скролла: cull + prepend в одном проходе не должны драться за `scrollTop`. Держать **одну** точку записи (`correctScroll`), как сейчас.
- Якорь hold-from-top должен переживать ресайз медиа (картинка догрузилась) — лучше якориться на ноду по id, а не на абсолютный `scrollTop`.
- Mock-чаты (без id и без реального loadNewer) не должны попадать под обрезку — гейтить по `isRealChat`.
