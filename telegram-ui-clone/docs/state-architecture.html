Два контура связи (важно для рефакторинга):

  • RPC (invoke/result) — все managers, запрос-ответ. Воркер прозрачно кэширует (SlicedArray для истории, Map для peers/media). UI кэш не видит.
  • Realtime (emit/on) — realtimeBridge — единственный «писатель» в chatsStore + дублирует в uiEvents. RT.newMessage/read/presence/typing → стор и шину; RT.ack/edit/delete/pin/reaction → только шину (их слушает ConversationView).
Ключевой вывод: chatsStore хранит только то, что нужно сайдбару (диалоги, presence, typing, me). Вся per-chat «начинка» (история, пины, card, счётчики комментов, online-сет) живёт локально в ConversationView — отсюда 28 useState + 18 useRef + 22 useEffect.

4 Принцип: НЕ всё в zustand

КорзинаЧтоКуда
Локальный useStateэфемерный UI компонента (корзина A)остаётся в компоненте
Кастомный хукимперативная DOM/scroll-машинерия (D), realtime-подпискихук, инкапсулирующий рефы+эффекты (это не «глобальный стейт»)
zustand сторпереживает >1 компонента, серверный кэш по чату, нужна гранулярная подписка (B, C)стор + селектор-хуки
Запихивать якоря меню в глобальный стор — антипаттерн (лишняя индирекция). Большой выигрыш по ререндерам мы уже взяли мемоизацией; zustand-декомпозиция здесь про поддерживаемость + правильную гранулярность подписок, а не про скорость печати.

5 Целевая архитектура

Новые сторы

  • conversationStore — per-chat серверный кэш, ключ = chatId. Поля: pins, card, commentCounts, onlineIds. Экшены loadCard/loadPins/… + апдейты от RT (pin) централизованно. Убирает из CV ~5 стейтов и ~5 эффектов.
  • selectionStore (опц.) — selected:Set, toggle/clear. Маленький, per-chat.
  • uiStore (опц., осторожно) — только оверлеи, открываемые из разных мест (lightbox/viewerMedia/discussion/call). Остальные меню — локально.

Слой селектор-хуков над chatsStore

Вместо разбросанных инлайн-стрелок, все с useShallow для массивов/объектов:

useMe() · useMeId() · useDialog(chatId) · useDialogs() · usePresence(peerId)
useTyping(chatId) · useMuted(chatId) · usePeerReadSeq(chatId)
// и для нового стора:
usePins(chatId) · useChatCard(chatId) · useCommentCounts(chatId) · useOnlineCount(chatId)

Кастомные хуки, выносимые из CV

useConversationScroll(...)   // корзина D: atBottom, restore, ResizeObserver, jump, ladder, player-offset
useChatRealtime(chatId, win) // uiEvents: ack/new/edit/delete/pin + markRead
useChatMeta(chatId)          // фасад над conversationStore (card/pins/counts/online)
Итог: ConversationView ужимается с ~2560 строк до оркестратора, который дёргает 3 хука и рендерит header/feed/composer.

6 План миграции по фазам

От безрискового к глубокому.

0

риск ~0  → единая точка, готовит почву
1
Вынести useConversationScroll и useChatRealtime из CV (без zustand).
низкий *  → −15 рефов/эффектов из CV
2
conversationStore (pins/card/commentCounts/online) + селектор-хуки; вырезать стейт+эффекты из CV.
средний  → per-chat данные шарятся, гранулярные подписки
3
опц.: selectionStore, uiStore для оверлеев; settings.tsx Context → zustand + persist.
низкий  → меньше Context, persist бесплатно
4
опц.: влить useMessageWindow в conversationStore.
высокий  → единый источник истории
* Фаза 1 трогает scroll-машинерию — самое хрупкое место (atBottomRef/restore/jump). Поэтому выносим её в изолированный хук и прогоняем сценарии (джамп к сообщению, подгрузка вверх/вниз, плеер, ладдер) через Playwright, как делали раньше.
Рекомендация: начать с Фазы 0 → 1 (чистый, проверяемый выигрыш и резкое ужатие CV), conversationStore (Фаза 2) — следом.

7 Паттерны zustand, которые внедрим

  • Селектор-хуки вместо инлайн-стрелок (DRY, единое место, типобезопасно).
  • useShallow для селекторов, возвращающих объект/массив (иначе ререндер на каждый set).
  • Slices pattern (combine/композиция) — если conversationStore разрастётся.
  • persist middleware для settings/voicePlayed (заменит ручной localStorage).
  • Атомарные селекторы — подписываемся на минимальный кусок (usePins не тянет card).
